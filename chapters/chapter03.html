3.1 팀 버너스 리의 비전

2001년 Scientific American에 발표된 팀 버너스 리의 논문 "The Semantic Web"은 웹의 미래에 대한 혁명적인 비전을 제시했습니다.

"나는 웹이 인간뿐만 아니라 기계도 이해할 수 있는 데이터의 웹이 되기를 꿈꿉니다. 언젠가는 컴퓨터가 웹의 모든 데이터를 분석할 수 있게 될 것입니다."
- 팀 버너스 리 (Tim Berners-Lee)

웹의 진화 단계

Web 1.0

1990-2000

읽기 전용 웹

  • 정적 HTML 페이지
  • 하이퍼링크로 연결
  • 일방향 정보 제공
예: 회사 홈페이지, 뉴스 사이트

Web 2.0

2000-2010

참여와 공유의 웹

  • 사용자 생성 콘텐츠
  • 소셜 네트워크
  • 동적 웹 애플리케이션
예: 페이스북, 유튜브, 위키피디아

Web 3.0

2010-현재

의미 기반 웹

  • 기계가 이해하는 데이터
  • 지능형 검색
  • 개인화된 경험
예: 구글 지식 그래프, Siri, Alexa

3.2 현재 웹의 한계

우리가 매일 사용하는 웹은 놀라운 성공을 거두었지만, 여전히 근본적인 한계를 가지고 있습니다.

문제점 1: 의미의 부재

현재 웹의 HTML

<h1>자바</h1>
<p>인도네시아의 섬</p>
<p>인구: 1억 4천만</p>

<h1>자바</h1>
<p>프로그래밍 언어</p>
<p>개발자: 제임스 고슬링</p>
⚠️ 컴퓨터는 두 '자바'가 다른 의미임을 모름

시맨틱 웹의 RDF

<rdf:Description rdf:about="자바_섬">
  <rdf:type>지리:섬</rdf:type>
  <지리:위치>인도네시아</지리:위치>
  <지리:인구>140000000</지리:인구>
</rdf:Description>

<rdf:Description rdf:about="자바_언어">
  <rdf:type>컴퓨터:프로그래밍언어</rdf:type>
  <컴퓨터:개발자>제임스_고슬링</컴퓨터:개발자>
</rdf:Description>
✅ 명확한 의미 구분과 관계 표현

문제점 2: 정보 사일로

쇼핑몰 A
• 상품명: 갤럭시 S24
• 가격: 1,200,000원
• 재고: 있음
쇼핑몰 B
• 제품: Galaxy S24
• 판매가: 120만원
• 상태: 판매중
쇼핑몰 C
• Name: 삼성 S24
• Price: ₩1,200,000
• Stock: 10
문제점
  • 같은 제품을 다르게 표현
  • 통합 검색 불가능
  • 가격 비교 어려움
  • 데이터 연계 불가

문제점 3: 검색의 한계

현재 키워드 검색

🔍 "서울 이탈리안 레스토랑 예약 가능"

검색 엔진이 이해하지 못하는 것:

  • '서울'이 위치임을 모름
  • '이탈리안'이 음식 종류임을 모름
  • '예약 가능'의 의미를 모름
  • 현재 시간 기준 판단 불가

시맨틱 검색

SELECT ?restaurant WHERE {
  ?restaurant rdf:type :ItalianRestaurant .
  ?restaurant :location :Seoul .
  ?restaurant :hasAvailableTable ?table .
  ?table :availableAt ?time .
  FILTER(?time > NOW())
}

시맨틱 웹의 장점:

  • 의미 기반 정확한 검색
  • 실시간 데이터 연동
  • 추론을 통한 확장 검색
  • 다국어 자동 처리

3.3 시맨틱 웹의 기술 스택

시맨틱 웹은 여러 계층의 기술들이 유기적으로 연결된 스택 구조로 이루어져 있습니다.

사용자 인터페이스 & 애플리케이션

지능형 에이전트, 시맨틱 검색 엔진

신뢰 (Trust)

전자서명, 평판 시스템

증명 (Proof)

논리적 증명, 검증

논리 (Logic)

추론 규칙, SWRL

온톨로지 (Ontology)

OWL, 어휘 정의

스키마 (Schema)

RDFS, 기본 어휘

데이터 (Data)

RDF, 트리플

기반 기술

URI, Unicode, XML

각 계층의 역할

1. URI (Uniform Resource Identifier)

웹 상의 모든 자원을 고유하게 식별

예시:
• http://example.org/people/홍길동
• http://dbpedia.org/resource/Seoul
• http://schema.org/Person

2. RDF (Resource Description Framework)

주어-술어-목적어 형태의 트리플로 정보 표현

홍길동
근무처
삼성전자

3. RDFS (RDF Schema)

클래스와 속성의 기본 어휘 제공

rdfs:Class      # 클래스 정의
rdfs:subClassOf # 상하위 관계
rdfs:domain     # 속성의 주어 범위
rdfs:range      # 속성의 목적어 범위

4. OWL (Web Ontology Language)

풍부한 의미 표현을 위한 온톨로지 언어

  • 복잡한 클래스 정의
  • 속성 특성 (대칭, 이행 등)
  • 카디널리티 제약
  • 논리적 추론 지원

3.4 시맨틱 웹에서 온톨로지의 역할

온톨로지는 시맨틱 웹의 핵심으로, 데이터에 의미를 부여하고 기계가 이해할 수 있게 만듭니다.

온톨로지의 주요 기능

📖

공통 어휘 제공

도메인의 개념과 관계를 표준화된 방식으로 정의

예: Schema.org는 웹 콘텐츠를 위한 공통 어휘 제공
🔗

데이터 통합

서로 다른 소스의 데이터를 의미적으로 연결

예: 병원 시스템과 보험사 시스템의 데이터 통합
🧠

지능적 추론

명시되지 않은 정보를 논리적으로 도출

예: "포유류는 따뜻한 피를 가진다" → "개는 따뜻한 피를 가진다"
🔍

의미 기반 검색

단순 키워드가 아닌 의미를 이해하는 검색

예: "심장 전문의"로 검색 시 "순환기내과 의사"도 결과에 포함

3.5 링크드 데이터 (Linked Data)

링크드 데이터는 시맨틱 웹의 실현을 위한 실천적 방법론입니다.

링크드 데이터 원칙

1

URI 사용

사물의 이름으로 URI를 사용하라

❌ "서울"
✅ http://dbpedia.org/resource/Seoul
2

HTTP URI

사람들이 찾을 수 있도록 HTTP URI를 사용하라

❌ urn:seoul:korea
✅ http://ko.dbpedia.org/resource/서울
3

유용한 정보 제공

URI를 찾았을 때 RDF와 SPARQL로 유용한 정보를 제공하라

RDF/XML, JSON-LD, Turtle 형식으로 데이터 제공
4

다른 URI 링크

더 많은 것을 발견할 수 있도록 다른 URI를 포함하라

서울 → 대한민국 → 아시아 → ...

링크드 오픈 데이터 클라우드

160억+ 트리플
5억+ 링크

🎯 실습 과제

나만의 링크드 데이터 만들기

좋아하는 주제를 선택하여 간단한 링크드 데이터를 설계해보세요:

Step 1: 주제 선택

예: 좋아하는 영화, 음악, 책, 여행지 등

Step 2: 주요 개체 식별

예: 영화 제목, 감독, 배우, 장르 등

Step 3: URI 설계

http://mydata.org/movie/기생충
http://mydata.org/person/봉준호
http://mydata.org/genre/스릴러

Step 4: 트리플 작성

<http://mydata.org/movie/기생충>
    rdf:type :Movie ;
    :director <http://mydata.org/person/봉준호> ;
    :genre <http://mydata.org/genre/스릴러> ;
    :releaseYear "2019" .

Step 5: 외부 데이터 연결

<http://mydata.org/movie/기생충>
    owl:sameAs <http://dbpedia.org/resource/Parasite_(2019_film)> .
힌트: DBpedia나 Wikidata에서 URI를 찾아 연결하면 더 풍부한 데이터가 됩니다!
