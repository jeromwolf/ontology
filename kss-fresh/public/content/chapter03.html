3.1 현재 웹의 한계와 도전과제

우리가 사용하는 현재의 웹(Web 2.0)은 놀라운 성공을 거두었지만, 정보의 의미를 이해하지 못한다는 근본적인 한계를 가지고 있습니다.

3.1.1 문법적 웹 vs 의미적 웹

문법적 웹 (현재)

특징
  • HTML 태그로 문서 구조 표현
  • 키워드 기반 검색
  • 사람을 위한 표현
  • 링크는 단순 참조
예시: 현재 웹의 표현

<div class="product">
    <h2>iPhone 13</h2>
    <p>가격: 1,250,000원</p>
    <p>제조사: Apple</p>
</div>

⚠️ 컴퓨터는 이것이 '제품'이라는 것을 이해하지 못함

의미적 웹 (시맨틱 웹)

특징
  • 온톨로지로 의미 정의
  • 개념 기반 검색
  • 기계가 이해 가능
  • 의미있는 관계 표현
예시: 시맨틱 웹의 표현

@prefix product: <http://example.org/product/> .
@prefix schema: <http://schema.org/> .

product:iPhone13 a schema:Product ;
    schema:name "iPhone 13" ;
    schema:price [
        a schema:PriceSpecification ;
        schema:price "1250000" ;
        schema:priceCurrency "KRW"
    ] ;
    schema:manufacturer [
        a schema:Organization ;
        schema:name "Apple Inc."
    ] .

✓ 컴퓨터가 제품, 가격, 제조사의 의미를 정확히 이해

3.1.2 현재 웹의 주요 문제점

1. 의미 모호성 (Ambiguity)

같은 단어가 문맥에 따라 다른 의미를 가짐

"Java"의 의미:
  • ☕ 커피 종류
  • 💻 프로그래밍 언어
  • 🏝️ 인도네시아의 섬
"Apple"의 의미:
  • 🍎 과일
  • 📱 기술 회사
  • 🎵 레코드 레이블

2. 데이터 사일로 (Data Silos)

각 웹사이트가 독립적인 데이터 섬

쇼핑몰 A

제품명: "갤럭시 S23"

분류: "스마트폰"

쇼핑몰 B

상품명: "Galaxy S23"

카테고리: "휴대폰"

리뷰 사이트

모델: "삼성 S23"

종류: "모바일기기"

💔 같은 제품인지 자동으로 연결할 수 없음

3. 단순한 링크 구조

링크가 관계의 의미를 표현하지 못함

현재의 링크
<a href="tim-berners-lee.html">팀 버너스 리</a>

→ 단순히 "연결됨"만 표현

의미있는 링크 (필요)
:WWW :inventedBy :TimBernersLee

→ "발명했다"는 관계 명시

4. 자동화의 한계

복잡한 작업을 자동화하기 어려움

시나리오: "다음 주 도쿄 출장 준비"

현재 (수동):

  1. 항공권 사이트에서 검색
  2. 호텔 예약 사이트 방문
  3. 환율 정보 확인
  4. 날씨 예보 확인
  5. 비자 요구사항 검색

😓 각 단계를 사람이 직접 수행해야 함

실습: 현재 웹의 한계 체험하기

과제: 특정 조건의 제품 찾기

다음 조건을 모두 만족하는 노트북을 찾아보세요:

  • 가격: 150만원 이하
  • 무게: 1.5kg 이하
  • 화면: 14인치 이상
  • 메모리: 16GB 이상
  • 보증기간: 2년 이상
현재 웹에서의 과정:
  1. 여러 쇼핑몰을 각각 방문
  2. 각 사이트의 다른 필터 인터페이스 사용
  3. 스펙 비교를 위해 수동으로 정보 수집
  4. 보증기간은 각 제품 상세페이지에서 확인
예상 소요 시간: 30분~1시간
시맨틱 웹에서의 해결:

SELECT ?laptop ?name ?price ?weight ?screen ?memory ?warranty
WHERE {
    ?laptop a :Laptop ;
            :hasPrice ?price ;
            :hasWeight ?weight ;
            :hasScreenSize ?screen ;
            :hasMemory ?memory ;
            :hasWarranty ?warranty ;
            :hasName ?name .
    
    FILTER (?price <= 1500000)
    FILTER (?weight <= 1.5)
    FILTER (?screen >= 14)
    FILTER (?memory >= 16)
    FILTER (?warranty >= 24)
}
ORDER BY ?price
예상 소요 시간: 1초 미만

3.2 팀 버너스 리의 시맨틱 웹 비전

월드 와이드 웹의 창시자 팀 버너스 리(Tim Berners-Lee)는 웹이 단순히 문서를 연결하는 것을 넘어서 데이터와 의미를 연결하는 거대한 지식 네트워크가 되기를 꿈꿨습니다.

Sir Tim Berners-Lee

월드 와이드 웹의 창시자

W3C (World Wide Web Consortium) 설립자

"나는 웹이 인간뿐만 아니라 기계도 이해할 수 있는 데이터의 웹이 되기를 꿈꿉니다. 언젠가는 컴퓨터가 웹의 모든 데이터를 분석할 수 있게 될 것입니다."

3.2.1 2001년 Scientific American 논문

📄 "The Semantic Web"

Tim Berners-Lee, James Hendler, Ora Lassila

Scientific American, May 2001

논문의 핵심 시나리오: Lucy와 Pete의 이야기

Lucy의 어머니가 정기적인 물리치료가 필요한 상황. Lucy와 그녀의 오빠 Pete는 어머니를 위한 적절한 치료 서비스를 찾아야 합니다.

1. 요구사항 입력

Lucy가 시맨틱 웹 에이전트에 입력:

  • 치료 종류: 물리치료
  • 보험: 어머니의 건강보험 정보
  • 위치: 집에서 20마일 이내
  • 평점: 우수 이상
  • 시간: 평일 오전
2. 자동 검색과 추론

에이전트의 작업:

  • 의료 서비스 온톨로지 검색
  • 보험사 데이터베이스와 연동
  • 지리 정보 시스템 활용
  • 리뷰 사이트 데이터 통합
  • 교통 정보 확인
3. 최적화된 결과

제시된 옵션:

  • Dr. Smith - 15마일, 평점 4.8
  • City Rehab Center - 12마일, 평점 4.5
  • 각 옵션의 장단점 분석
  • Pete의 일정과 조율 가능한 시간
4. 자동 예약

선택 후 자동 처리:

  • 병원 예약 시스템과 연동
  • 보험 사전 승인 처리
  • 캘린더에 일정 추가
  • 교통 경로 안내 설정

3.2.2 시맨틱 웹의 핵심 원칙

1. 개방성 (Openness)

누구나 참여하고 기여할 수 있는 열린 웹

  • 개방형 표준 사용
  • 독점 기술 배제
  • 상호운용성 보장

2. 분산성 (Distribution)

중앙 집중식이 아닌 분산된 지식

  • 각자 자신의 데이터 관리
  • P2P 방식의 연결
  • 단일 실패점 없음

3. 계층성 (Layering)

단계적 기술 스택 구축

  • 하위 계층의 안정성
  • 상위 계층의 확장성
  • 점진적 도입 가능

4. 신뢰성 (Trust)

검증 가능한 정보와 출처

  • 디지털 서명
  • 출처 추적
  • 신뢰 네트워크

3.2.3 웹의 진화: Web 1.0에서 Web 3.0으로

Web 1.0

1990-2000
읽기 전용 웹 (Read-Only Web)

정적 HTML 페이지

하이퍼링크 연결

일방향 정보 전달

대표 사례:
  • Yahoo 디렉토리
  • 초기 Amazon
  • 정적 홈페이지
핵심 기술:
HTML, HTTP, URL

Web 2.0

2000-2020
참여와 공유의 웹 (Read-Write Web)

사용자 참여

소셜 네트워크

동적 콘텐츠

대표 사례:
  • Facebook, Twitter
  • YouTube, Instagram
  • Wikipedia, Blog
핵심 기술:
AJAX, REST API, RSS, JSON

Web 3.0

2020-미래
의미와 지능의 웹 (Read-Write-Execute Web)

기계 이해

AI 통합

분산화

대표 사례:
  • Knowledge Graph
  • Voice Assistants
  • Blockchain, DeFi
핵심 기술:
RDF, OWL, SPARQL, AI/ML, Blockchain

인터랙티브 비교: 검색의 진화

검색 시나리오: "2시간 이내 완성 가능한 이탈리안 요리 레시피"

Web 1.0 검색
  1. Yahoo 디렉토리 → 요리 → 이탈리안
  2. 각 사이트 수동 방문
  3. 레시피 하나씩 확인
  4. 시간 정보 직접 계산

결과: 매우 제한적, 시간 소모적

Web 2.0 검색
  1. Google: "이탈리안 요리 레시피 2시간"
  2. 키워드 매칭 결과
  3. 사용자 리뷰 참고
  4. 여러 결과 비교

결과: 많은 결과, 수동 필터링 필요

Web 3.0 검색
  1. 자연어 이해: "2시간 이내 이탈리안"
  2. 온톨로지 기반 추론
  3. 재료 가용성 확인
  4. 개인화된 추천

결과: 정확한 맞춤형 레시피

3.3 시맨틱 웹 기술 스택 (Layer Cake)

시맨틱 웹은 여러 기술이 층층이 쌓여 구성되는 "Layer Cake" 아키텍처를 따릅니다. 각 계층은 하위 계층을 기반으로 하며, 상위로 갈수록 더 복잡하고 지능적인 기능을 제공합니다.

시맨틱 웹 Layer Cake

Trust (신뢰)

디지털 서명, 평판 시스템

Proof (증명)

논리적 증명, 추론 검증

Logic (논리)

추론 규칙, 일차 논리

Ontology (온톨로지)

OWL - 복잡한 지식 표현

owl:Class, owl:ObjectProperty, owl:DatatypeProperty

RDFS (RDF Schema)

기본 어휘와 스키마 정의

rdfs:Class, rdfs:subClassOf, rdfs:domain, rdfs:range

RDF (Resource Description Framework)

트리플 기반 데이터 모델

Subject - Predicate - Object

XML + Namespaces

구조화된 데이터 교환 형식

<rdf:RDF xmlns:rdf="...">

Unicode + URI

전역적 식별자와 문자 인코딩

http://example.org/resource#item

3.3.1 각 계층의 상세 설명

1. Unicode + URI 계층

역할

시맨틱 웹의 기초로, 전 세계 어떤 리소스든 고유하게 식별하고 표현

핵심 구성요소
  • URI (Uniform Resource Identifier)
    • 모든 리소스의 고유 식별자
    • 예: http://dbpedia.org/resource/Seoul
  • IRI (Internationalized Resource Identifier)
    • 다국어 지원 URI
    • 예: http://example.org/리소스/서울
  • Unicode
    • 전 세계 모든 문자 체계 지원
    • 다국어 온톨로지 구축 가능
예제: URI 활용

# 한국어 리소스 URI 예제
@prefix kr: <http://example.kr/resource/> .
@prefix foaf: <http://xmlns.com/foaf/0.1/> .

kr:서울특별시 a kr:광역시 ;
    foaf:name "서울특별시"@ko ;
    foaf:name "Seoul"@en ;
    kr:인구 "9,733,509" .

2. XML + Namespaces 계층

역할

구조화된 데이터 표현과 서로 다른 어휘의 혼용

핵심 기능
  • 계층적 데이터 구조 표현
  • 네임스페이스로 어휘 충돌 방지
  • RDF/XML 직렬화 지원
예제: RDF/XML

<?xml version="1.0" encoding="UTF-8"?>
<rdf:RDF 
    xmlns:rdf="http://www.w3.org/1999/02/22-rdf-syntax-ns#"
    xmlns:foaf="http://xmlns.com/foaf/0.1/"
    xmlns:dc="http://purl.org/dc/elements/1.1/">
    
    <rdf:Description rdf:about="http://example.org/book/1">
        <dc:title>온톨로지 입문</dc:title>
        <dc:creator>
            <foaf:Person>
                <foaf:name>김철수</foaf:name>
            </foaf:Person>
        </dc:creator>
    </rdf:Description>
</rdf:RDF>

3. RDF 계층

역할

시맨틱 웹의 핵심 데이터 모델 제공

트리플 모델
주어 (Subject)

설명하려는 리소스

:Leonardo_da_Vinci
→
술어 (Predicate)

속성이나 관계

:painted
→
목적어 (Object)

값이나 다른 리소스

:Mona_Lisa
RDF 직렬화 형식
Turtle

@prefix : <http://example.org/> .
:김철수 :works_for :삼성전자 ;
       :age 30 .
JSON-LD

{
  "@context": "http://example.org/",
  "@id": "김철수",
  "works_for": {"@id": "삼성전자"},
  "age": 30
}
N-Triples

<http://example.org/김철수> <http://example.org/works_for> <http://example.org/삼성전자> .
<http://example.org/김철수> <http://example.org/age> "30"^^<http://www.w3.org/2001/XMLSchema#integer> .

4. RDFS 계층

역할

RDF에 스키마 기능을 추가하여 기본적인 온톨로지 구축

주요 구성요소
클래스 정의
  • rdfs:Class - 클래스 선언
  • rdfs:subClassOf - 상하위 관계
  • rdf:type - 인스턴스 관계
속성 정의
  • rdf:Property - 속성 선언
  • rdfs:domain - 속성의 주어 제한
  • rdfs:range - 속성의 목적어 제한
문서화
  • rdfs:label - 사람이 읽을 레이블
  • rdfs:comment - 설명
  • rdfs:seeAlso - 관련 리소스
예제: RDFS로 간단한 스키마 정의

@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix : <http://example.org/vehicles#> .

# 클래스 계층구조
:Vehicle rdf:type rdfs:Class ;
    rdfs:label "차량"@ko ;
    rdfs:comment "모든 운송수단의 상위 클래스"@ko .

:Car rdfs:subClassOf :Vehicle ;
    rdfs:label "자동차"@ko .

:ElectricCar rdfs:subClassOf :Car ;
    rdfs:label "전기자동차"@ko .

# 속성 정의
:manufacturedBy rdf:type rdf:Property ;
    rdfs:domain :Vehicle ;
    rdfs:range :Company ;
    rdfs:label "제조사"@ko .

:maxSpeed rdf:type rdf:Property ;
    rdfs:domain :Vehicle ;
    rdfs:range xsd:integer ;

3.5 Linked Data와 5-Star Open Data

Linked Data는 시맨틱 웹의 실질적인 구현 방법론으로, 웹상의 데이터를 서로 연결하여 거대한 글로벌 데이터 공간을 만드는 것을 목표로 합니다.

3.5.1 Linked Data의 4원칙

1

URI로 사물 이름 짓기

모든 리소스에 HTTP URI를 부여하여 웹에서 식별 가능하게 함

http://dbpedia.org/resource/Seoul http://data.gov.kr/resource/city/11000
2

HTTP URI 사용

사람과 기계가 조회할 수 있도록 HTTP 프로토콜 사용

❌ urn:isbn:978-89-123-4567-8

✅ http://example.org/book/978-89-123-4567-8

3

유용한 정보 제공

URI 접근 시 RDF, SPARQL 등 표준 형식으로 정보 제공

Content Negotiation 예시

GET /resource/Seoul HTTP/1.1
Host: example.org
Accept: text/html → HTML 페이지
Accept: application/rdf+xml → RDF/XML
Accept: text/turtle → Turtle
4

다른 URI로의 링크 포함

관련 정보를 발견할 수 있도록 다른 데이터셋으로의 링크 제공


<http://data.gov.kr/Seoul>
    owl:sameAs <http://dbpedia.org/resource/Seoul> ;
    owl:sameAs <http://www.geonames.org/1835848> ;
    rdfs:seeAlso <http://en.wikipedia.org/wiki/Seoul> .

실습: 나만의 Linked Data 만들기

한국 관광지 Linked Data 구축

Step 1: URI 스키마 설계

기본 네임스페이스: http://korea-tourism.kr/

URI 패턴:
- 도시: http://korea-tourism.kr/city/{name}
- 명소: http://korea-tourism.kr/attraction/{id}
- 축제: http://korea-tourism.kr/festival/{id}
- 음식: http://korea-tourism.kr/food/{name}
Step 2: 데이터 모델링

@prefix kt: <http://korea-tourism.kr/> .
@prefix geo: <http://www.w3.org/2003/01/geo/wgs84_pos#> .
@prefix schema: <http://schema.org/> .
@prefix dbpedia: <http://dbpedia.org/resource/> .

kt:city/seoul a schema:City ;
    schema:name "서울특별시"@ko , "Seoul"@en ;
    geo:lat "37.5665" ;
    geo:long "126.9780" ;
    schema:description "대한민국의 수도"@ko ;
    owl:sameAs dbpedia:Seoul ;
    kt:hasAttraction kt:attraction/gyeongbokgung ,
                     kt:attraction/n-seoul-tower .
Step 3: 외부 데이터 연결

kt:attraction/gyeongbokgung a schema:TouristAttraction ;
    schema:name "경복궁"@ko , "Gyeongbokgung Palace"@en ;
    schema:description "조선왕조의 정궁"@ko ;
    geo:lat "37.5796" ;
    geo:long "126.9770" ;
    schema:openingHours "Mo-Su 09:00-18:00" ;
    schema:isAccessibleForFree false ;
    owl:sameAs dbpedia:Gyeongbokgung ;
    owl:sameAs <http://www.wikidata.org/entity/Q483501> ;
    schema:image <http://korea-tourism.kr/images/gyeongbokgung.jpg> ;
    kt:nearbyFood kt:food/kimchi ,
                  kt:food/bulgogi .
Step 4: SPARQL 엔드포인트 제공

# 서울의 모든 관광명소 검색
PREFIX kt: <http://korea-tourism.kr/>
PREFIX schema: <http://schema.org/>

SELECT ?attraction ?name ?description
WHERE {
    kt:city/seoul kt:hasAttraction ?attraction .
    ?attraction schema:name ?name ;
                schema:description ?description .
    FILTER (lang(?name) = "ko")
}
