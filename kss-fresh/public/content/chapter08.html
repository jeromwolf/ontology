8.2 Protégé로 첫 온톨로지 만들기

🎯 실습 프로젝트: 스마트 홈 온톨로지

IoT 기기, 가전제품, 공간, 사용자를 표현하는 스마트 홈 온톨로지를 단계별로 구축해봅니다.

📝 Step 1: 새 온톨로지 생성

  1. File → New 클릭
  2. Ontology IRI 입력: http://example.org/smart-home
  3. Ontology Version IRI: http://example.org/smart-home/1.0
  4. Format: RDF/XML 선택

💡 Pro Tip

IRI(Internationalized Resource Identifier)는 온톨로지의 고유 식별자입니다. 일반적으로 도메인명/프로젝트명 형식을 사용합니다.

🏗️ Step 2: 클래스 계층 구조 설계

클래스 생성 방법

  1. Classes 탭 선택
  2. owl:Thing 선택 → 오른쪽 클릭 → "Add subclass"
  3. 클래스 이름 입력

스마트 홈 클래스 구조

owl:Thing
├── Device
│   ├── Sensor
│   │   ├── TemperatureSensor
│   │   ├── MotionSensor
│   │   └── LightSensor
│   ├── Actuator
│   │   ├── Light
│   │   ├── Heater
│   │   └── AirConditioner
│   └── Controller
│       ├── Hub
│       └── SmartSpeaker
├── Room
│   ├── LivingRoom
│   ├── Bedroom
│   ├── Kitchen
│   └── Bathroom
├── User
│   ├── Owner
│   └── Guest
└── Automation
    ├── Rule
    └── Scene

Disjoint Classes 설정

서로 격커지는 클래스들을 지정합니다:

  1. Device, Room, User, Automation 선택
  2. Edit → Make disjoint classes

🔗 Step 3: Object Properties 정의

객체 간 관계 설정

1. locatedIn (위치 관계)
Domain: Device
Range: Room
Characteristics: Functional (하나의 기기는 하나의 방에만 위치)
2. controls (제어 관계)
Domain: Controller
Range: Device
Inverse: controlledBy
3. hasOwner (소유 관계)
Domain: Device
Range: User
Characteristics: Functional
4. triggers (트리거 관계)
Domain: Sensor
Range: Automation

Property Chain 예제

locatedIn ◦ locatedIn → locatedIn

(예: 센서가 방에 있고, 방이 집에 있으면, 센서는 집에 있다)

📊 Step 4: Data Properties 정의

데이터 속성 예제

hasTemperature
Domain: TemperatureSensor
Range: xsd:float
Functional: true
hasBrightness
Domain: Light
Range: xsd:integer[0..100]
Functional: true
deviceName
Domain: Device
Range: xsd:string
Functional: true
isActive
Domain: Device ∪ Automation
Range: xsd:boolean
Functional: true

👥 Step 5: Individuals 생성

개체 생성 예제

1. 온도 센서 개체
Individual: livingRoomTempSensor
Type: TemperatureSensor
Assertions:
  - locatedIn livingRoom
  - hasTemperature 23.5
  - deviceName "Living Room Temperature Sensor"
  - isActive true
2. 사용자 개체
Individual: johnDoe
Type: Owner
Assertions:
  - hasDevice livingRoomTempSensor
  - hasDevice smartHub1
3. 자동화 규칙
Individual: nightModeRule
Type: Rule
Assertions:
  - triggeredBy motionSensor1
  - controls bedroomLight
  - isActive true

8.3 추론 엔진과 지식 검증

🧠 추론의 힘

Protégé의 추론 엔진은 명시적으로 입력하지 않은 사실들을 자동으로 추론해냅니다.

🔍 주요 추론 엔진 비교

추론 엔진 특징 성능 지원 OWL 사용 시나리오
HermiT 가장 정확, 완전성 보장 중간 OWL 2 DL 일반적인 온톨로지
Pellet 균형 잡힌 성능 좋음 OWL 2 DL 중간 규모 프로젝트
ELK 초고속 성능 매우 빠름 OWL 2 EL 대규모 의료 온톨로지
Fact++ C++ 기반, 빠른 속도 빠름 OWL 2 DL 복잡한 계층 구조

🎯 추론 실습

Step 1: 추론 엔진 시작

  1. Reasoner 메뉴 → HermiT 선택
  2. Reasoner → Start reasoner 클릭
  3. 하단 상태바에 "Reasoner Active" 확인

Step 2: 추론 결과 확인

🔸 클래스 계층 추론
# 정의: SmartLight ⊆ Light ∩ ∃controls.Controller
# 추론: SmartLight는 Light의 하위 클래스
🔸 개체 분류
# 사실: philipsHue1 controls smartHub1
# 정의: SmartLight ≡ Light ∩ ∃controlledBy.Controller
# 추론: philipsHue1 ∈ SmartLight
🔸 속성 추론
# 사실: sensor1 locatedIn bedroom1, bedroom1 locatedIn house1
# 정의: locatedIn은 transitive
# 추론: sensor1 locatedIn house1

Step 3: 일관성 검증

모순 감지 예시
  1. 설정: Light와 Sensor를 disjoint로 설정
  2. 실수: device1을 Light와 Sensor 모두에 할당
  3. 결과: Inconsistent ontology 오류
🔧 해결 방법
  • Reasoner → Explain inference 사용
  • 충돌하는 axiom 확인
  • 잘못된 assertion 수정

🔍 DL Query 사용법

Description Logic 쿼리로 복잡한 검색을 수행할 수 있습니다.

DL Query 예제

1. 특정 방에 있는 모든 센서
Sensor and locatedIn value livingRoom
2. 컨트롤러가 없는 기기
Device and not (controlledBy some Controller)
3. 온도가 25도 이상인 센서
TemperatureSensor and hasTemperature some xsd:float[>= 25.0]
4. 활성화된 자동화 규칙
Rule and isActive value true

💡 DL Query 팁

  • and: 교집합 (모든 조건 만족)
  • or: 합집합 (하나 이상 만족)
  • not: 부정
  • some: 존재 양화사 (∃)
  • only: 전칭 양화사 (∀)
  • value: 특정 값

🔍 SPARQL 통합

SPARQL Query 탭 활성화

  1. Window → Tabs → SPARQL Query
  2. SPARQL 탭이 표시됨
Protégé에서 SPARQL 예제
PREFIX : 
PREFIX rdf: 

# 모든 온도 센서와 현재 온도
SELECT ?sensor ?temp ?room
WHERE {
    ?sensor rdf:type :TemperatureSensor ;
            :hasTemperature ?temp ;
            :locatedIn ?room .
    FILTER(?temp > 25)
}
ORDER BY DESC(?temp)

8.5 종합 프로젝트: 의료 진단 온톨로지

🏥 프로젝트 개요

증상, 질병, 검사, 치료를 표현하는 간단한 의료 진단 온톨로지를 구축합니다.

📋 Step 1: 클래스 구조 설계

MedicalEntity
├── ClinicalEntity
│   ├── Disease
│   │   ├── InfectiousDisease
│   │   │   ├── BacterialInfection
│   │   │   └── ViralInfection
│   │   └── ChronicDisease
│   │       ├── Diabetes
│   │       └── Hypertension
│   ├── Symptom
│   │   ├── Pain
│   │   │   ├── Headache
│   │   │   └── ChestPain
│   │   ├── Fever
│   │   └── Fatigue
│   └── Treatment
│       ├── Medication
│       │   ├── Antibiotic
│       │   └── Painkiller
│       └── Procedure
│           ├── Surgery
│           └── Therapy
├── DiagnosticTest
│   ├── LaboratoryTest
│   │   ├── BloodTest
│   │   └── UrineTest
│   └── ImagingTest
│       ├── XRay
│       └── MRI
└── Patient
    ├── InPatient
    └── OutPatient

🔗 Step 2: 속성 정의

Object Properties

# 진단 관련
hasSymptom
    Domain: Disease
    Range: Symptom
    
showsSymptom
    Domain: Patient
    Range: Symptom
    InverseOf: symptomOf

hasDiagnosis
    Domain: Patient
    Range: Disease
    Functional: false (환자는 여러 질병을 가질 수 있음)

# 검사 관련
requiresTest
    Domain: Disease
    Range: DiagnosticTest
    
underwentTest
    Domain: Patient
    Range: DiagnosticTest

# 치료 관련
treatedWith
    Domain: Disease
    Range: Treatment
    
receivesTreatment
    Domain: Patient
    Range: Treatment

# 인과 관계
causedBy
    Domain: Disease
    Range: MedicalEntity
    Transitive: true

Data Properties

hasAge
    Domain: Patient
    Range: xsd:integer[0..150]
    Functional: true

hasSeverity
    Domain: Symptom ∪ Disease
    Range: {"mild", "moderate", "severe"}
    Functional: true

hasOnsetDate
    Domain: Disease ∪ Symptom
    Range: xsd:date
    Functional: true

hasDuration
    Domain: Symptom
    Range: xsd:string
    Example: "2 days", "1 week"

testResult
    Domain: DiagnosticTest
    Range: xsd:string

🧮 Step 3: 복합 클래스 정의

# 고혈압 환자
HypertensionPatient ≡ 
    Patient ∩ (hasDiagnosis some Hypertension)

# 중증 환자
SeverePatient ≡ 
    Patient ∩ (hasDiagnosis some (Disease ∩ hasSeverity value "severe"))

# 항생제 치료가 필요한 질병
AntibioticTreatableDisease ≡ 
    Disease ∩ (treatedWith some Antibiotic)

# 긴급 환자
EmergencyPatient ≡ 
    Patient ∩ (showsSymptom some (Symptom ∩ hasSeverity value "severe"))
         ∩ (showsSymptom min 3 Symptom)

📜 Step 4: SWRL 규칙 추가

진단 규칙

# 발열 + 두통 + 목 통증 → 독감 가능성
Patient(?p) ∧ showsSymptom(?p, ?s1) ∧ Fever(?s1) ∧
showsSymptom(?p, ?s2) ∧ Headache(?s2) ∧
showsSymptom(?p, ?s3) ∧ SoreThroat(?s3)
→ hasDiagnosis(?p, Flu)

# 고혈압 + 당뇨 → 심혈관 위험
Patient(?p) ∧ hasDiagnosis(?p, ?d1) ∧ Hypertension(?d1) ∧
hasDiagnosis(?p, ?d2) ∧ Diabetes(?d2)
→ hasRisk(?p, "CardiovascularDisease")

# 나이 기반 검사 추천
Patient(?p) ∧ hasAge(?p, ?age) ∧ swrlb:greaterThan(?age, 50)
→ requiresTest(?p, ColonoscopyScreening)

🧠 Step 5: 추론 실행과 결과

시나리오: 환자 진단

  1. 입력 데이터: 
    Individual: patient1
Types: Patient
Facts:
    hasAge 55
    showsSymptom fever1
    showsSymptom headache1
    showsSymptom fatigue1

Individual: fever1
Types: Fever
Facts:
    hasSeverity "high"
    hasOnsetDate "2024-01-15"
  2. 추론 결과:
    • patient1 ∈ EmergencyPatient (중증 증상)
    • patient1 hasDiagnosis Flu (SWRL 규칙)
    • patient1 requiresTest ColonoscopyScreening (나이 기반)

8.3 추론 규칙과 메커니즘

기본 추론 규칙

1. 포함관계 추론

A ⊑ B ∧ B ⊑ C → A ⊑ C

예시:

Dog ⊑ Mammal ∧ Mammal ⊑ Animal

→ Dog ⊑ Animal

2. 속성 전파

P ⊑ Q ∧ (x, y) ∈ P → (x, y) ∈ Q

예시:

hasMother ⊑ hasParent

John hasMother Mary → John hasParent Mary

3. 도메인/레인지 추론

domain(P, C) ∧ (x, y) ∈ P → x ∈ C

예시:

domain(teaches, Teacher)

Kim teaches Math → Kim ∈ Teacher

4. 이행성 규칙

Trans(P) ∧ P(x,y) ∧ P(y,z) → P(x,z)

예시:

Trans(ancestorOf)

A ancestorOf B ∧ B ancestorOf C

→ A ancestorOf C

추론 알고리즘

Tableau Algorithm

가장 널리 사용되는 결정 절차

  • ABox와 TBox 일관성 검사
  • 개체 분류
  • 완전성 보장

Resolution

일차 논리 기반 추론

  • 규칙 기반 추론
  • 호른 절 처리 효율적
  • 불완전할 수 있음

Rete Algorithm

규칙 엔진에서 사용

  • 패턴 매칭 최적화
  • 증분적 추론
  • 대규모 규칙 처리

HermiT

Oxford 대학에서 개발한 OWL 추론기

장점
  • OWL 2 DL 완전 지원
  • 높은 정확성
  • 우수한 오류 설명
단점
  • 대규모 온톨로지에서 느림
  • 메모리 사용량 높음
적합한 경우: 정확성이 중요한 중소규모 온톨로지

Pellet

Java 기반 오픈소스 OWL DL 추론기

장점
  • 증분적 추론 지원
  • SPARQL-DL 질의
  • 좋은 문서화
단점
  • 개발 중단 (유지보수만)
  • 최신 OWL 2 기능 제한적
적합한 경우: 안정성이 중요한 프로젝트

ELK

OWL 2 EL 프로파일 특화 추론기

장점
  • 매우 빠른 속도
  • 대규모 온톨로지 처리
  • 병렬 처리 지원
단점
  • OWL EL만 지원
  • 표현력 제한
적합한 경우: 대규모 분류 체계 (예: SNOMED CT)

Fact++

C++ 기반 고성능 추론기

장점
  • 빠른 처리 속도
  • 메모리 효율적
  • OWL 2 지원
단점
  • 제한적인 API
  • 오류 메시지 빈약
적합한 경우: 성능이 중요한 응용 프로그램

8.5 추론 실습

예제: 가족 관계 추론

온톨로지 정의

# 클래스 정의
Person ⊑ Thing
Male ⊑ Person
Female ⊑ Person
Parent ≡ Person ⊓ ∃hasChild.Person
Father ≡ Parent ⊓ Male
Mother ≡ Parent ⊓ Female

# 속성 정의
hasParent inverse hasChild
hasFather ⊑ hasParent, range Father
hasMother ⊑ hasParent, range Mother
hasBrother ≡ hasSibling ○ Male

# 개체와 관계
John : Male
Mary : Female
Tom : Person
John hasChild Tom
Mary hasChild Tom

추론 결과

자동 분류:
  • John : Father (Male이고 hasChild 관계 있음)
  • Mary : Mother (Female이고 hasChild 관계 있음)
역관계 추론:
  • Tom hasParent John
  • Tom hasParent Mary
  • Tom hasFather John
  • Tom hasMother Mary
일관성 검증:
  • Male ⊓ Female ≡ ⊥ (상호 배타적)
  • Father ⊓ Mother ≡ ⊥ (자동 유도됨)

8.6 추론 성능 최적화

성능 향상 기법

1. 온톨로지 모듈화

큰 온톨로지를 작은 모듈로 분할

  • 관련 개념끼리 그룹화
  • 필요한 부분만 로드
  • 독립적 추론 가능
// 모듈 임포트
import: <http://example.org/person-module>
import: <http://example.org/location-module>

2. 프로파일 선택

용도에 맞는 OWL 프로파일 사용

  • OWL EL: 대규모 분류
  • OWL QL: 데이터베이스 연동
  • OWL RL: 규칙 기반 시스템

3. 증분적 추론

변경된 부분만 재계산

  • 추론 결과 캐싱
  • 변경 추적
  • 부분 업데이트

4. 추론 깊이 제한

필요한 수준까지만 추론

  • 추론 체인 길이 제한
  • 특정 속성 제외
  • 근사 추론 활용

추론기 선택 가이드

시나리오 추천 추론기 이유
의료 온톨로지 (SNOMED CT) ELK 대규모 + EL 프로파일
연구용 소규모 온톨로지 HermiT 정확성 + 디버깅
실시간 웹 애플리케이션 Pellet/Fact++ 균형잡힌 성능
규칙 기반 시스템 Drools + Pellet 규칙 엔진 통합

🎯 실습 과제

과제 1: 대학 온톨로지 추론

다음 규칙을 포함하는 온톨로지를 만들고 추론 결과를 확인하세요:

  1. 학부생은 학부 과목만 수강 가능
  2. 대학원생은 모든 과목 수강 가능
  3. 교수는 자신이 가르치는 과목을 수강할 수 없음
  4. 필수과목을 모두 수강한 학생은 졸업 가능

과제 2: 추론기 성능 비교

동일한 온톨로지를 다른 추론기로 처리하고 비교하세요:

  1. 추론 시간 측정
  2. 메모리 사용량 확인
  3. 추론 결과 차이 분석
